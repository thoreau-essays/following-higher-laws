## Design Context

### Users
DH scholars, educators, and students — primarily at University of Idaho and peer institutions — building long-form scholarly essays that incorporate primary source items from digital collections. Many are non-technical: they choose the template, write content, and populate metadata rather than design. Usage is in offices, classrooms, and at home; primarily desktop/laptop. Student readers may access on mobile. The tool also serves instructors teaching digital scholarship and online literacy, where the output of the tool itself demonstrates good digital publishing practice.

### Brand Personality
Three words: **thoughtful, approachable, useful**

The interface should feel like a well-designed university press publication brought to the web — considered and serious without being cold or intimidating. It should communicate that scholars with real things to say can present their work professionally without being designers. The design disappears into the reading experience.

### Aesthetic Direction
Tufte-inspired: wide margins with sidenotes, restrained decoration, text column constrained for optimal reading length (~60-65ch), generous whitespace. The reference aesthetic is Edward Tufte's physical books and their web interpretation (tufte.css). Think MIT Press or Princeton University Press web presence — refined, academic, light-touch.

**Light mode.** Scholarly reading happens in offices, classrooms, and at desks during the day.

**Anti-references:**
- Generic Bootstrap site with nav-blue, white background, system fonts
- Corporate SaaS / dashboard feel
- Overly decorative or magazine-y (should feel like a book, not Vox)
- AI-default aesthetics: gradient text, glassmorphism, cyan-on-dark, purple-blue gradients

**Differentiator:** CB-Essay is one of very few digital humanities publishing frameworks that works out-of-the-box as a static site. The design should be the clearest visual signal that this is a considered, disciplined platform — not just another Jekyll theme.

### Design Principles
1. **Typography leads.** The reading experience is the product. Font choice, spacing, and rhythm carry the design; decoration does not.
2. **Elegant restraint.** No decoration that doesn't serve the scholarship. White space and margin notes are the design features.
3. **Tufte column layout.** Main text column constrained to ~633px/60-65ch. Margin notes float right into the generous gutter. Hierarchy through spacing, not visual noise.
4. **Configurable defaults.** Looks professionally published without any customization. Flexible enough for scholars to adapt to their specific project without breaking anything.
5. **Print parity.** Web and PDF/print are equals — both receive equal design attention. The print output should feel like a well-typeset academic document, not browser print.

### Font Guidance
Per the font selection procedure: brand words are "thoughtful, approachable, useful." Reflex picks (Fraunces, Newsreader, Lora, Cormorant, Crimson) are all on the ban list.

**Default (no CDN required):** Georgia — the project owner has confirmed Georgia as the right default for the DH audience, which often avoids Google Fonts CDN for privacy/institutional reasons. Georgia is a genuinely good scholarly serif. Keep it.

**Suggested upgrade path (for documentation):** Three tiers to offer in the docs:
- *Tier 1 — No CDN:* Georgia + system sans (Helvetica/Arial) — the default, works everywhere
- *Tier 2 — Google Fonts:* Spectral (body, Production Type) + Hanken Grotesk (UI/headings). Warm, screen-optimized, not reflex picks. Font CDN: `https://fonts.googleapis.com/css2?family=Spectral:ital,wght@0,300;0,400;0,600;1,300;1,400&family=Hanken+Grotesk:wght@400;500;600&display=swap`
- *Tier 3 — Self-hosted:* Vollkorn (body, open license) + similar sans — for institutions with strict CDN policies

### Color Guidance
CB-Essay ships with a neutral black-and-white default palette. Individual projects add their own accent color via `link-color` in `_data/theme.yml`.

Background: warm off-white — `oklch(97.5% 0.003 90)`. Not pure white; the slight warmth reads as paper.

Text: near-black — `oklch(13% 0.004 90)`. Not pure black; tint toward warmth.

Borders and dividers: `oklch(87% 0.003 90)` — neutral warm gray.

Secondary / muted text: `oklch(42% 0.004 90)`.

Links: driven by `link-color` in `theme.yml` — default `#3d7b53`, user-configurable per project.

Avoid: pure `#000` or `#fff`. Avoid Bootstrap blue `#0d6efd` hardcoded anywhere. Do not bake project-specific accent colors into the template CSS — leave those as theme.yml settings.

### Print System Notes
The Paged.js print system is now functional. Key architectural decisions made during the fix:

**What works:**
- Images sized correctly — `feature-w-25/50/75/100` and `feature-float-start/end` classes added to `_print-paged.scss` (they're defined in `_base.scss`/`cb.scss` which doesn't load on print pages)
- Control bar visible on screen, hidden cleanly at print time via `beforeprint`/`afterprint` events
- No double-render: content is moved into a detached `DocumentFragment` before `preview()` is called; the empty `#print-content` shell is immediately hidden
- Cover page on page 1: `min-height` is now injected per-format by JS (letter: 8.5in, A4: 9in, 6×9: 6.25in) — was 9in hardcoded which overflowed the letter content area (8.75in)
- Essay filtering (`?essays=` param): non-matching chapters are removed from the DOM entirely before Paged.js runs — `display:none` caused Paged.js to generate blank pages for each hidden element
- Code blocks: `break-inside: avoid` prevents page splits; `white-space: pre-wrap` + `overflow-wrap: break-word` prevents long lines from overflowing the page column
- No "Unable to layout item: renderTo" error: caused by `<figure>` → `<div>` DOM replacement mid-layout; fixed by adding `aria-label` to figures instead

**Key Paged.js behaviors to remember:**
- `PrintMedia` handler hoists ALL `@media print {}` rules to global scope — anything in `@media print` applies unconditionally during screen preview too
- `removeStyles()` removes all `<link>` and `<style>` elements except those with `[data-pagedjs-ignore]` or `[media~='screen']` — then re-processes them through Paged.js's own CSS engine
- `display:none` elements are still walked by Paged.js's layout engine and can generate blank pages — always remove from DOM instead
- Inline `element.style.setProperty(prop, val, 'important')` overrides any stylesheet `!important` due to higher cascade specificity

**Known remaining edge cases:**
- `.print-title-page { min-height: 8in }` still overflows 6×9 content area (6.75in) — same fix needed as cover (format-aware JS injection)
- Code blocks longer than a full page will still split — unavoidable, but short/medium blocks are protected
- `break-after: right` on the cover page creates a blank page 2 between cover and TOC (intentional book publishing behavior, but may surprise authors)

**Files:**
- `_layouts/print.html` — template + all JS (pre-render prep, Paged.js init, A11y handler)
- `_sass/_print-paged.scss` → compiled to `assets/css/print.css` — print-only stylesheet, never loaded on live site
- `pages/print/book.md` — the book print page (`book: true` triggers book mode in print.html)
- `pages/print/index.md` — print hub UI (format selector, individual essay links, book builder)
